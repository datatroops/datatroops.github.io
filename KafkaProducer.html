Understanding Kafka Producer, Consumer using GraphDSL

Building real-time data pipelines and streaming applications often uses the Apache Kafka distributed event streaming framework. It enables handling of data streams that is high-throughput, fault-tolerant, and scalable. We will discuss the terms "Kafka producer" and "Kafka consumer" in this section, as well as give a brief overview of Alpakka Kafka, a well-liked streaming library for integrating Kafka with Akka Streams. We will also discuss about Akka streams’ powerful tool, GraphDSL. Akka Streams and Akka Kafka libraries to interact with Apache kafka, a distributed message broker.

Kafka Producer : (ref : akka.kafka.scaladsl.Producer)

Producer is responsible for publishing the data to Kafka topics. Producers send records as key-value pair, where keys are optional and used to partition data, while values contain the actual data to be stored in Kafka. The Kafka producer is designed to be highly efficient, and it is possible to batch records and send them in parallel to achieve high throughput.

Let’s deep dive into the code startProducer file :

  1. Importing the required libraries and packages : It imports akka-related packages and the ExecutionContextExecutor to manage asynchronous operations. 
    // Package declaration
package io.datatroops
                                    
// Import required libraries and packages
import akka.NotUsed
import akka.actor.ActorSystem
import akka.stream.Materializer.createMaterializer
import akka.stream.scaladsl.{Broadcast, GraphDSL, RunnableGraph, Source}
import akka.stream.{ClosedShape, Materializer}
import io.datatroops.producer.AlpakkaKafkaProducer
import io.datatroops.serialisers.StringSerializer.stringSerializer
import org.apache.kafka.clients.producer.ProducerRecord
import scala.concurrent.ExecutionContextExecutor

  1. Initializing the ActorSystem, ExecutionContextExecutor, Materializer: An ActorSystem is initialized, which is required for creating actors and managing their lifecycle. The ExecutionContextExecutor is used to execute asynchronous operations, and the Materializer is created using createMaterializer to execute Akka Streams components.
    // Initialize ActorSystem, ExecutionContextExecutor, Materializer
implicit val system: ActorSystem = ActorSystem("kafkaProducerSystem")
implicit val ec: ExecutionContextExecutor = system.dispatcher
implicit val materializer: Materializer = createMaterializer(system)
  1. Declaring a Source:

    A source(messageSource) is created using Akka Streams’ Source that generates messages as strings from integer 40 to 45 (As an example). The map function converts each integer to its string representation.

    // Create a Source that generates messages as strings from integers 40 to 45
val messageSource: Source[String, NotUsed] = Source(40 to 45).map(_.toString)
  1. AlpakkaKafkaProducer Initialisation:

    The AlpakkaKafkaProducer is initialized, which is a custom class representing the kafka producer logic. It provides methods like getProducer, getKafkaSink, getStatusSink for different parts of producer.

    // Initialize the AlpakkaKafkaProducer with key and value serializers
val alpakkaKafkaProducer = AlpakkaKafkaProducer[String, String]
val producer = alpakkaKafkaProducer.getProducer
val kafkaSink = alpakkaKafkaProducer.getKafkaSink
val kafkaStatusSink = alpakkaKafkaProducer.getStatusSink
  1. Transformation: Transformation is done to the data received as messageSource, incremented the numbers by 10.
    // Transformation
val incrementToFlow = Flow[String].map(message => ((message.toInt) + 10).toString)
  1. Runnable Graph (Producer): A graph is created using Akka Streams’ GraphDSL . The graph includes a Broadcast stage, which takes single input and outputs to two different outlets. The message from the messageSource are broadcasted to both the kafkaSink and kafkaStatusSink.
    // Create a graph for the producer, broadcasting messages to kafkaSink and kafkaStatusSink
val producerGraph = RunnableGraph.fromGraph(GraphDSL.create() { implicit builder =>
    import GraphDSL.Implicits._
    val broadcast = builder.add(Broadcast[String](1))
    messageSource ~> incrementToFlow ~> broadcast
    broadcast.out(0) ~> producer ~> kafkaSink
    ClosedShape
}).run

Let’s check AlpakkaKafkaProducer and check the declarations from it,

  1. AlpakkaKafkaProducer case class:

    The AlpakkaKafkaProducer class is defined as a case class that represents the kafka producer configuration and logic. It is parameterised with types K and V, representing key and value types of the Kafka records.

    // Case class Declaration of AlpakkaKafkaProducer
case class AlpakkaKafkaProducer[K, V]()(implicit keySerializer: Serializer[K],
valueSerializer: Serializer[V],system: ActorSystem ) extends Producer[K, V]
  1. BootstrapServers and Topic declaration: The bootstrapServers and topic are set to specify the kafka broker details and the topic to which messages will be sent.
    // Configure the Kafka broker details
override val bootstrapServers: String = "localhost:9092"
override val topic: String = "quickstart-events"
  1. ProducerSettings: The ProducerSettings method is defined to get the Kafka producer settings with the provided key and value serializers. It configures the producer with the specified bootstrap servers.
    // Get the producer settings with provided serializers
override def producerSettings(implicit keySerializer: Serializer[K], valueSerializer: Serializer[V])
                     : ProducerSettings[K, V] = ProducerSettings(system, keySerializer, valueSerializer)
                                        .withBootstrapServers(bootstrapServers)
  1. getKafkaSink: The getKafkaSink is defined to get a Kafka sink, which is used to send messages to the Kafka topic. It uses the plainSink method from Akka Kafka’s Producer object, passing the producer settings.
    // Get a Kafka sink to send messages
override val getKafkaSink: Sink[ProducerRecord[K, V], Future[Done]] = Producer.plainSink(producerSettings)
  1. getProducer: The getProducer method is defined to get a flow that transforms input messages of type V to Kafka ProducerRecords with keys of type K. The flow uses the map function to create a new ProducerRecord for each message with the specifies topic and message.
    // Get a flow to transform messages to ProducerRecords
override def getProducer: Flow[V, ProducerRecord[K, V], NotUsed] = Flow[V].map(msg => new ProducerRecord[K, V](topic, msg))
  1. getStatusSink: The getStatusSink is defined to get a sink that logs the status of sent messages. It uses the Sink.foreach to print the value of each sent ProducerRecord to the console.
    override def getStatusSink: Sink[ProducerRecord[K, V], Future[Done]] = 
Sink.foreach[ProducerRecord[K, V]](value => println(s"${value.value()}"))

WorkFlow:

  • Initialize Akka Streams components (ActorSystem, Materializer).
  • Create a source that generates messages as strings.
  • Initialize the AlpakkaKafkaProducer with key and value serializers.
  • Create a graph to process messages from the Source and broadcast them to both kafka sinks.
  • Run the graph to start producing messages to the Kafka topic.

Kafka Consumer : (ref : akka.kafka.scaladsl.Consumer)

Kafka Consumer is responsible for subscribing to Kafka topics and processing incoming records. Consumers can be part of a consumer group, allowing them to share the load of processing data from different partitions of a topic. Each record is delivered to one consumer within the group.

Understanding of code snippet - StartConsumer: The StartConsumer code snippet demonstrates how to create a kafka consumer using Akka Streams and Alpakka Kafka. Let’s break down the key components.

  1. Importing Necessary Libraries and Packages: We need to import the necessary libraries and packages similar to StartProducer code.
    // Import required libraries and packages
  package io.datatroops
  import akka.actor.ActorSystem
  import akka.kafka.Subscriptions
  import akka.kafka.scaladsl.Consumer
  import akka.stream.Materializer.createMaterializer
  import scala.concurrent.ExecutionContextExecutor

  1. Initializing the ActorSystem, ExecutionContextExecutor, Materializer: An ActorSystem is initialized, which is required for creating actors and managing their lifecycle. The ExecutionContextExecutor is used to execute asynchronous operations, and the Materializer is created using createMaterializer to execute Akka Streams components.
    // Initialize ActorSystem, ExecutionContextExecutor, and Materializer
implicit val system: ActorSystem = ActorSystem("kafkaProducerSystem")
implicit val ec: ExecutionContextExecutor = system.dispatcher
implicit val materializer: Materializer = createMaterializer(system)
  1. Initialize AlpakkaKafkaConsumer with key and value deserializers: The AlpakkaKafkaConsumeris initialized, which is a custom class representing the kafka consumer logic. It provides methods like consumerSettings, dataEventFlow, and printStoredData.
    // Initialize AlpakkaKafkaConsumer with key and value deserializers
val consumer = AlpakkaKafkaConsumer[String, String]()
val consumerSetting = consumer.consumerSettings
val printDataEvent = consumer.dataEventFlow()
val printStoreData = consumer.printStoredData()
  1. Creating a runnable graph for the consumer, processing data from kafka:
    // Create a graph for the consumer, processing data from Kafka 
val consumerGraph = RunnableGraph.fromGraph(GraphDSL.create() { implicit builder =>
    import GraphDSL.Implicits._
    val broadcast = builder.add(Broadcast[ConsumerRecord[String, String]](3))
    val subscriptions = Subscriptions.topics(topic)
    Consumer.plainSource(consumerSetting, subscriptions) ~> broadcast

    // Step 4.1: Print the received data events and ignore the result
    broadcast.out(1) ~> printDataEvent ~> Sink.ignore

    // Step 4.2: Error handling for messages received from Kafka
    val errorHandler = builder.add(Sink.foreach[Throwable](exception => println(s"Faced Error ${exception.getMessage}")))

    // Step 4.3: Log and throw an exception for any error during message processing
    broadcast.out(0).map(error => throw new RuntimeException(s"Failed to receive the message from kafka topic ${error.toString}")) ~> errorHandler

    // Step 4.4: Store the received data
    val dataStore = builder.add(printStoreData)
    broadcast.out(2) ~> dataStore
    ClosedShape
  }).run

    A Graph is created using Akka Streams’ GraphDSL . The Graph involves multiple stages for processing data from Kafka.

    • The data received from Kafka is broadcasted to an outlet that flows into the printDataEvent flow. This flow processes and prints the received data events using Sink.ignore, effectively ignoring the output.
    • Another outlet of the broadcast is mapped to the errorHandler sink. The sink handles errors (throwable exceptions) that may occur during message processing. Any exception is printed to the console.
    • The third outlet of the broadcast is connected to the dataStore sink, which processes and stores the received data.

    The Runnable Graph is executed by calling run(), and the consumer starts processing and storing messages from the Kafka topic.

    Certainly! Let's explain the other file AlpakkaKafkaConsumer:

    1. The AlpakkaKafkaConsumer class is defined as a case class that represents the kafka consumer configuration and logic. It is parameterized with types K and V, representing the key and value types of the Kafka records.
      case class AlpakkaKafkaConsumer[K, V]()(implicit keyDeserializer: Deserializer[K], valueDeserializer: Deserializer[V], system: ActorSystem)
                                      extends Consumer[K, V]
    1. The groupId, bootstrapServer, and topic are set to specify the Kafka consumer group, broker details, and the topic from which messages will be consumed.
      // Configure the Kafka broker details and consumer group
override val groupId: String = "group_id"
override val bootstrapServer: String = "localhost:9092"
override val topic: String = "quickstart-events"
    1. The consumerSettings method is defined to get the Kafka consumer settings with the provided key and value deserializers. It configures the consumer with the specified bootstrap servers, consumer group, and offset reset policy. 
      // Get the consumer settings with provided deserializers
override def consumerSettings: ConsumerSettings[K, V] = ConsumerSettings(system, keyDeserializer, valueDeserializer)
                .withBootstrapServers(bootstrapServer)
                .withGroupId(groupId)
                .withProperty(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, "earliest")
    1. The dataEventFlow method is defined to get a flow that processes the received ConsumerRecords and extracts the value (message) from each record. The map function is used to extract the value, and the mapMaterializedValue is used to return NotUsed as the materialized value. 
      // Get a flow to print the received data events
def dataEventFlow(): Flow[ConsumerRecord[K, V], V, NotUsed] = {
                      Flow[ConsumerRecord[K, V]].map { consumerRecord =>
                        consumerRecord.value()
                      }
                        .mapMaterializedValue(_ => NotUsed)
    1. The printStoredData method is defined to get a sink that prints the received data (messages) to the console. It uses Sink.foreach to print the value of each received ConsumerRecord to the console.
      // Get a sink to print the stored data
def printStoredData(): Sink[ConsumerRecord[K, V], _] = {
  Sink.foreach[ConsumerRecord[K, V]](data => println(s"${data.value()}"))
}
    1. The kafkaSource method is defined to get a source that reads data from Kafka. It uses Akka Kafka's plainSource method, passing the consumer settings and the topic to subscribe to. It returns a source with ConsumerRecords and Consumer.Control, which can be used for controlling the consumer, such as shutting it down gracefully.
      // Get a source to read data from Kafka
def kafkaSource(): Source[ConsumerRecord[K, V], Consumer.Control] = {
  Consumer.plainSource(consumerSettings, Subscriptions.topics(topic))
}

    In summary, the AlpakkaKafkaProducer and AlpakkaKafkaConsumer are custom classes that encapsulate the Kafka producer and consumer logic, respectively. They provide methods to get the required Akka Streams components (such as sources, flows, and sinks) to interact with Kafka topics. These classes abstract away the complexities of setting up the producer and consumer settings and provide a cleaner and more reusable way to work with Kafka within Akka Streams.

    The GraphDSL approach provides more flexibility for designing complex data flow patterns and enables you to build sophisticated processing logic. It is particularly beneficial when dealing with advanced stream processing scenarios, such as conditional branching, fan-in, fan-out, and custom components. In Summary, while a simple approach without GraphDSL is suitable for linear pipelines, using GraphDSL offers enhanced composability and customizability, making it a preferred choice when building complex stream processing graphs with more intricate data flow patterns.
